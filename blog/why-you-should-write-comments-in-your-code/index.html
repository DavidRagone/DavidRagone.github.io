Why You Should Write Comments In Your Code

July 5, 2017    programming principles

Why you should write comments in your code

Aka, self-documenting code and other programming myths.

Especially in the ruby community, there’s a strong animosity to commenting or documenting code. The usual retort is something to the effect of “comments are future lies”, referencing how they can easily become an incorrect representation of what the code does.

In my experience, this both ignores an important aspect of the job, and misstates the purpose of comments. To briefly address the first point, I think it’s quite possible and reasonable to expect comments to stay up to date. It simply needs to be a standard expectation, set and reinforced by code reviews.

Tests answer “what”

In addition to proving correctness and protecting against regressions, tests explain what the code does. Whether a feature spec (it "sends a password reset email") or a unit spec (it "returns first and last name, separated by a space"), I know that I can learn about code by reading the tests. This is why I often start code reviews by looking at the tests; they help me understand what the code is doing.

Code shows how

Code is the implementation. The email is sent via DelayedJob, asynchronously, on the high priority queue. There’s some systems there to understand, but these are details meant to fulfill a spec. There’s a lot that’s important to writing good code, and a lot one can learn by reading it. In the end, it mostly answers how things are happening.

Comments tell you why

Neither the code nor the tests can illuminate the fact that this particular feature was added because of the new big bank client who required extra security. Code is unlikely to state the possible temporariness (because the bank only signed a 6 month deal to start), and the specs won’t emphasize the criticality of it.

This is where you need comments. They provide critical context to other developers (including future you).
